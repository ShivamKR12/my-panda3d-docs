201 // The WindowRenderer class records the stages of the pipeline that each
202 // thread (including the main thread, a.k.a. "app") should process, and the
203 // list of windows for each stage.
204
205 // There is one WindowRenderer instance for app, and another instance for
206 // each thread (the thread-specific WindowRenderers are actually instances
207 // of RenderThread, below, which inherits from WindowRenderer).
208
209 // The idea is that each window is associated with one or more
210 // WindowRenderer objects, according to the threads in which its rendering
211 // tasks (window, cull, and draw) are divided into.
212
213 // The "window" task is responsible for doing any updates to the window
214 // itself, such as size and placement, and is wholly responsible for any API
215 // calls to the windowing system itself, unrelated to OpenGL-type calls.
216 // This is normally done in app (the design of X-Windows is such that all X
217 // calls must be issued in the same thread).
218
219 // The "cull" task is responsible for crawling through the scene graph and
220 // discovering all of the Geoms that are within the viewing frustum. It
221 // assembles all such Geoms, along with their computed net state and
222 // transform, in a linked list of CullableObjects, which it stores for the
223 // "draw" task, next.
224
225 // The "draw" task is responsible for walking through the list of
226 // CullableObjects recorded by the cull task, and issuing the appropriate
227 // graphics commands to draw them.
228
229 // There is an additional task, not often used, called "cdraw". This task,
230 // if activated, will crawl through the scene graph and issue graphics
231 // commands immediately, as each Geom is discovered. It is only rarely used
232 // because it cannot perform sorting beyond basic scene graph order, making
233 // it less useful than a separate cull and draw task.
234
235 // It is possible for all three of the normal tasks: window, cull, and draw,
236 // to be handled by the same thread. This is the normal, single-threaded
237 // model: all tasks are handled by the app thread. In this case, the window
238 // will be added to _app's _window, _cull, and _draw lists.
239
240/*
241 * On the other hand, a window's tasks may also be distributed among as many
242 * as three threads. For instance, if the window is listed on _app's _window
243 * list, but on thread A's _cull list, and thread B's _draw list, then the
244 * window task will be handled in the app thread, while the cull task will be
245 * handled by thread A, and the draw task will be handled (in parallel) by
246 * thread B. (In order for this to work, it will be necessary that thread A
247 * and B are configured to view different stages of the graphics pipeline.
248 * This is a more advanced topic than there is room to discuss in this
249 * comment.)
250 */
251
252 // Manipulation of the various window lists in each WindowRenderer object is
253 // always performed in the app thread. The auxiliary threads are slaves to
254 // the app thread, and they can only perform one of a handful of specified
255 // tasks, none of which includes adding or removing windows from its lists.
256 // The full set of tasks that a WindowRenderer may perform is enumerated in
257 // ThreadState, above; see RenderThread::thread_main().
258
259/*
260 * There is a pair of condition variables for each thread, _cv_start and
261 * _cv_done, that is used to synchronize requests made by app to a particular
262 * thread. The usual procedure to request a thread to perform a particular
263 * task is the following: the app thread waits on the thread's _cv_done
264 * variable, stores the value corresponding to the desired task in the
265 * thread's _thread_state value, then signals the thread's _cv_start variable.
266 * The thread, in turn, will perform its requested task, set its _thread_state
267 * to TS_wait, and signal _cv_done. See examples in the code, e.g.
268 * open_windows(), for more details on this process.
269 */
270
271 // It is of course not necessary to signal any threads in order to perform
272 // tasks listed in the _app WindowRenderer. For this object only, we simply
273 // call the appropriate methods on _app when we want the tasks to be
274 // performed.
